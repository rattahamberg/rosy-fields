# This is NOT the Next.js you know

This version has breaking changes ‚Äî APIs, conventions, and file structure may all differ from your training data. Read the relevant guide in `node_modules/next/dist/docs/` before writing any code. Heed deprecation notices.

Specific Next 16 things in THIS project:
- The middleware file is `proxy.ts` at the project root (NOT `middleware.ts`). It exports a `proxy` function and a `config`.
- `cookies()` and `headers()` from `next/headers` are async ‚Äî always `await`.
- `searchParams` and `params` page props are `Promise<...>`. Always destructure via `await`.
- `forbidden()` from `next/navigation` requires `experimental.authInterrupts: true` in `next.config.ts` (already enabled).

- App Router lives at the project root ‚Äî there's no `src/`. Path alias `@/*` ‚Üí `./*`.

# Project conventions

## Auth

- Session DAL: `lib/dal.ts` exposes:
  - `peekSession()` ‚Äî returns the session or `null`, never redirects. Use in decoration-only components like `<AdminEmail>` that should degrade quietly when there's no session.
  - `verifySession()` ‚Äî returns the session or `redirect("/login")`s. Builds on `peekSession`.
  - `getUser()` ‚Äî convenience: returns `session.user` or redirects.
- Admin DAL: `lib/admin/dal.ts` exposes `verifyAdmin()` and `getCurrentRole()`. **Always call `verifyAdmin()` at the top of every admin page** ‚Äî the layout-only check is insufficient under partial rendering.
- All DALs are wrapped in `react.cache()`, so repeated calls in a single render dedupe to one auth lookup per request.
- Non-admins hit `forbidden()` ‚Üí `app/forbidden.tsx` (NOT `notFound()`).

## Server Actions

Two patterns, picked by need:

- **`useActionState`-style** (`createHouseholdAction`, `renameHouseholdAction`): action returns `FormState = { ok: true; id?: string } | { ok: false; error: string }`. Use this when the form has user-typed field state worth preserving across validation errors.
- **Redirect-style** (`deleteHousehold`, `addMember`, `removeMember`): action accepts `FormData`, performs the mutation, then `redirect()`s. Use this for one-shot actions where there's nothing to preserve.

Every mutation MUST:
1. Call `await verifyAdmin()` first.
2. Wrap the DB write AND the audit log insert in one `db.transaction()` so they commit atomically.
3. Use the FK-violation catch (`PG_FK_VIOLATION = "23503"`) instead of pre-flight SELECT checks where possible (no TOCTOU).
4. `revalidatePath()` for any page that displays the affected data.
5. For redirect-style: use `safePath()` from `lib/safe-redirect.ts` on any user-supplied `redirectTo`.

## Audit log

`lib/admin/audit.ts` exposes `writeAudit(entry, options?)`. ALL admin mutations must write an audit row in the same transaction as the mutation. View-page audits use `after()` to avoid blocking response time. See **Audit context (`writeAudit`)** below for the call-site convention.

## Migrations

- Schema lives in `lib/db/schema.ts`.
- After editing schema: `npx drizzle-kit generate --name <description>`.
- Apply locally with `npm run migrate`. CI applies prod automatically on push to master.
- The `__migrations` table tracks SHA-256 of each file. **Never edit a migration file after it's been applied** ‚Äî `npm run migrate` will refuse to run if file content drifts from the stored hash.
- For hand-written DDL not generated by drizzle-kit (e.g. trigger definitions), add a manually-written `drizzle/000X_<description>.sql` file AND add the entry to `drizzle/meta/_journal.json`.

## Schema conventions

- All timestamp columns use `withTimezone: true` (timestamptz).
- `updatedAt` columns use `.$onUpdate(() => new Date())` so Drizzle auto-bumps them.
- All FK columns are indexed (the planner needs them for cascade/set-null operations).
- Trigram indexes (`gin (col gin_trgm_ops)`) on text columns that get `ILIKE` searched.
- Search inputs gate on `ADMIN_SEARCH_MIN_LENGTH = 3` because trigram needs ‚â•3 chars to use the index.

## Constants

`lib/admin/config.ts` is the single home for admin tunables (page sizes, picker limits, name length bounds, search minimum, error display caps). Don't introduce new magic numbers in admin files ‚Äî add them there.

## Component layout

- Admin shared components: `app/admin/_components/` (underscore = private folder, never routable). Import via the barrel: `import { AdminTable, Section, DataGrid, DetailHeader } from "@/app/admin/_components"`.
- The barrel **excludes** `<AdminEmail>` ‚Äî it's server-only (transitively imports `lib/db`) and re-exporting would leak `server-only` into client bundles via the error boundaries that import the barrel. Layout imports it directly from `./admin-email`.
- `<PrimaryButton>` lives at `app/_components/primary-button.tsx` (NOT inside `admin/_components`) because login + signup forms also use it. Import: `import { PrimaryButton } from "@/app/_components/primary-button"`.
- Co-locate single-use client components with the page that owns them (e.g. `app/admin/households/[id]/delete-household-form.tsx`).
- `<AdminTable headers={[...]}>` for any tabular admin data ‚Äî locks the outer markup so detail pages don't drift visually. Pass `null` in the headers array for unlabelled action columns.
- `<Section title>` for grouped content blocks on detail pages.
- `<DetailHeader>` for the title + back-link pattern shared across detail pages.
- `<DataGrid rows={[{ label, value }]}>` for label/value pairs (identity cards).
- `<PrimaryButton>` for the dark "submit" button style. Variants via `size="sm" | "md"`.

## Audit context (`writeAudit`)

- **Mutation actions**: resolve `net = await readNetworkContext()` BEFORE opening `db.transaction`, then pass `{ client: tx, net }` to `writeAudit`. The two-overload signature on `writeAudit` enforces this ‚Äî `client` requires `net` at the type level so the lock window never includes a header read.
- **View-page audits via `after()`**: just call `writeAudit(entry)` with no options. The fallback resolves network context internally; no transaction is open so the resolve is harmless.

## Tooling

- `npm run grant-admin -- <email> [--dry-run]` to promote a user to admin.
- `npm run migrate` to apply pending migrations.
- `npm run migrate:bootstrap` only on a brand-new database where the schema was applied out-of-band.
- `npm run migrate:rehash` after a deliberate hash-affecting edit (e.g. CRLF normalization, adding IF NOT EXISTS guards). Updates the stored sql_hash in `__migrations` without running anything. Same flag is available in CI via the workflow_dispatch input `rehash`.
- `scripts/setup-cron.sql` documents optional pg_cron retention/purge jobs (apply manually after enabling pg_cron in the Neon console).

## Error boundaries

- `app/admin/error.tsx`, `app/dashboard/error.tsx`, and `app/global-error.tsx` use `unstable_retry` (Next 16.2.0+, expect rename to `retry` later). They re-fetch and re-render the boundary's children rather than just re-rendering with stale data.
- Per-detail-page errors (`app/admin/users/[id]/error.tsx`, `app/admin/households/[id]/error.tsx`) wrap the shared `<AdminDetailError label logTag>` so a row-level failure doesn't unmount the admin shell.
